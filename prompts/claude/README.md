# Универсальный шаблон многоагентной промпт-архитектуры для Claude

Этот каталог — переносимый шаблон системы промптов для оркестрации нескольких
Claude-агентов над одним проектом. Он извлечён из работающей архитектуры (изначально
построенной для конкретного проекта) и де-специфицирован: все имена, идентификаторы,
хосты, даты и UUID заменены на плейсхолдеры `{{...}}`, а все внутренние пути
переписаны так, чтобы **корень этого каталога был корнем архитектуры** — то есть
шаблон можно скопировать в любой проект как есть, без переноса вложенности
`docs/agent-ref/...`.

---

## 1. Что это

Архитектура построена по принципу **«тонкое ядро + ленивые триггеры»**:

- **Ядро** — несколько небольших YAML-файлов, которые обязан прочитать *каждый*
  агент перед началом работы (`roles/common.yaml`, `roles/laws.yaml`, свой
  `roles/<role>.yaml`). Ядро маленькое и не меняется от задачи к задаче.
- **Триггеры** — остальное (карты серверов, операционные детали инструментов,
  специфика режимов) агент читает **только когда впервые собирается сделать
  действие**, к которому этот файл привязан. Ничего не грузится «на всякий
  случай» — это и есть «ленивость»: экономия контекста ценой обязательного
  правила «прочитай перед первым использованием».

### Роли

| Роль | Кто это | Что делает |
|---|---|---|
| `orchestrator` | верхнеуровневый агент, общается с пользователем | принимает задачу, решает, что делать, делегирует всё исполнение |
| `conscience` | адversarial-зеркало оркестратора | придирчиво критикует решение оркестратора, выдаёт только вердикт |
| `context_former` | сборщик контекста для одного шага | ищет факты, собирает самодостаточную задачу для coder+tester |
| `researcher` | read-only исследователь | изучает код/документацию, ничего не пишет |
| `coder` | исполнитель правки | редактирует один файл через editor lifecycle |
| `tester` | проверяющий | гоняет QA-гейт и тесты, выдаёт вердикт coder'у |
| `executor` | координатор runtime-исполнения замороженного плана | ведёт учёт попыток исполнения через plan-manager, оркестрирует пары coder+tester по атомарным шагам |

### Законы (`roles/laws.yaml`)

Общие для всех ролей стоячие правила с параметрами сессии
(`variables.file_access`, `variables.escalation`), которые пользователь/оркестратор
выставляет один раз в начале сессии:

- `mcp_only` — файловые операции только через MCP (если не включён `local`);
- `local_mode` — что меняется в профиле `local` (commit-after-edit и т.д.);
- `escalation_first_error` — стоп на первой инфра-ошибке;
- `reproduce_before_claiming` — не объявлять баг без чистого воспроизведения;
- `verify_claims_independent_model` — блокирующий claim субагента проверяется
  независимой моделью прежде чем на него реагировать;
- `editor_gate_no_bypass` — HARD-правило, QA-гейт редактора нельзя обходить
  никогда и ни в каком профиле;
- `ssh_repair` — ssh-доступ к серверу — привилегия только оркестратора;
- `delivery_checks` — обязательные проверки перед сдачей результата пользователю.

### Карты серверов (`servers/`) и операционные блоки (`ops/`)

`servers/*.yaml` — «тонкая карта» одного MCP-сервера: его live-идентификатор в
реестре, «жёсткие правила» (hard_rules) и таблица «триггер → какой `ops/*.yaml`
читать». Сам `ops/*.yaml` — детальная процедура + команды + gotchas + коды
ошибок для одного функционального блока этого сервера (например,
`ca-search.yaml` — блок поиска code-analysis-сервера).

### Режимы (`modes.yaml`)

Набор именованных «режимов работы» — какие `ops/*.yaml` подключаются
дополнительно к базовым триггерам `roles/tooling.yaml`, когда оркестратор в
задаче объявляет конкретный режим.

### Схема каталога

```text
.
├── CLAUDE.md                        # точка входа: роль ORCHESTRATOR + @imports
├── modes.yaml                       # карта режим -> триггеры (planning/analysis/refactoring)
├── roles/
│   ├── common.yaml                  # универсальные законы (зоны, эскалация, spawn-протокол)
│   ├── laws.yaml                    # стоячие законы с variables сессии
│   ├── tooling.yaml                 # механика инструментов (для tool-using ролей)
│   ├── tooling-rules.yaml           # "боевые" gotcha по инструментам
│   ├── coder-guide.yaml             # сводный мануал кодера (механика MCP-редактирования)
│   ├── orchestrator.yaml            # роль: оркестратор
│   ├── conscience.yaml              # роль: адversarial-вердикт
│   ├── context_former.yaml          # роль: сборщик контекста + спавнер coder/tester
│   ├── researcher.yaml              # роль: read-only исследователь
│   ├── coder.yaml                   # роль: кодер
│   ├── tester.yaml                  # роль: тестировщик
│   └── executor.yaml                # роль: runtime-исполнитель замороженного плана
├── servers/
│   ├── ca.yaml                      # карта code-analysis-сервера
│   ├── editor.yaml                  # карта ai-editor-сервера
│   ├── terminal.yaml                # карта терминал-сервера (sandbox + host_exec)
│   └── planmgr.yaml                 # карта plan-manager (глоссарий + методология + блоки)
└── ops/
    ├── ca-*.yaml                    # 6 файлов: search/analysis/refactoring-support/git/jobs-queue/fs-sandbox
    ├── editor-*.yaml                # 3 файла: edit-lifecycle/new-file/formats-addressing
    ├── planmgr-*.yaml               # 9 файлов: authoring/cascade/verification/graph/prompt/runtime/bugs/todo/project/export
    └── terminal-*.yaml              # 2 файла: sandbox/host-exec
```

Итого 38 файлов. Все внутренние пути в файлах — **относительно корня этого
каталога** (`roles/...`, `ops/...`, `servers/...`, `modes.yaml`), без
префикса вида `docs/agent-ref/`. Если вы кладёте архитектуру в проект под
другой вложенностью (например, `docs/agent-ref/`), **CLAUDE.md — единственное
место, требующее ручной правки путей** (`@roles/common.yaml` и т.д.
станет `@docs/agent-ref/roles/common.yaml`) — см. чеклист адаптации, пункт (г).

## 2. Как это работает

### Спавн-протокол

Каждый субагент, которого порождает оркестратор или `context_former`,
получает задачу, начинающуюся строго с:

> First read `roles/common.yaml` AND `roles/laws.yaml` and every file listed in
> `roles/<role>.yaml` `reads_first` (via Read or CA preview) — do NOT spawn a
> subagent to read. Then: `<task>`.

Это гарантирует, что субагент прочитал ядро **сам**, а не «со слов» родителя —
пересказ ядра в теле задачи не считается заменой чтения.

### Триггерный закон

`roles/tooling.yaml` (читают только tool-using роли: researcher, context_former,
coder, tester) формулирует правило: **перед первым использованием инструмента,
привязанного к триггеру, агент обязан прочитать файл-манул этого триггера хотя
бы один раз за сессию**. Это относится и к базовым триггерам (`rules`, `editor`,
`ca_search` и т.п.), и к триггерам режима (см. ниже) — они складываются, а не
подменяют друг друга.

### Режим объявляется в задаче

Оркестратор (или `context_former`) явно называет режим в тексте задачи
(например: «в режиме planning...» / «...mode: refactoring»). Без объявленного
режима субагент работает только с базовым набором триггеров `roles/tooling.yaml`.
Режим **добавляет** свои триггеры поверх базовых, никогда не заменяет их.
`modes.yaml` — единственное место, где перечислены все режимы; `header_rule` в
начале файла напоминает: режим говорит **когда**, `tooling.yaml` — **как**.

### Выбор модели по сложности задачи

- **mechanical / однократная механическая правка** → слабая модель (haiku);
- **standard / многошаговая работа** (researcher, context_former, tester,
  executor, большинство coder) → средняя модель (sonnet);
- **вердикты, аудит, независимая верификация** (conscience) → сильная модель
  (opus).

Слабую модель никогда не отправляют на задачу, требующую суждения — по опыту
исходного проекта она начинает галлюцинировать/додумывать под давлением.
Перепроверьте это наблюдение на своём стеке моделей, прежде чем полагаться на
него как на аксиому.

### Эскалация с первой инфра-ошибки

`escalation_first_error` (`roles/laws.yaml`): любая инфраструктурная ошибка
(таймаут, 5xx, потеря сессии, WAF 403) — это стоп, а не повод для ретраев или
обходных путей. Отчёт идёт вверх по цепочке `agent → orchestrator → user` с
{точным вызовом, дословной ошибкой, конкретным вопросом}. Единственное
исключение — простая VALIDATION_ERROR, самостоятельно исправимая через
`help(cmdname)`, и то один раз.

### Закон языка (`roles/laws.yaml` → `language`)

RUS используется ТОЛЬКО для общения с пользователем (user-facing сообщения,
вопросы, отчёты, адресованные пользователю). Всё остальное — ENG: код,
комментарии, commit-сообщения, файлы промптов/контрактов, задачи и отчёты
между агентами, записи багов/todo, логи. Субагент никогда не пишет по-русски
в артефакты — перевод для пользователя делает оркестратор при финальном
отчёте. Это тот же принцип «одна роль — одна зона», применённый к языку:
внутренний слой архитектуры говорит на одном языке (ENG) независимо от того,
на каком языке ведёт диалог с пользователем оркестратор.

### Принцип need-to-know (зоны)

`roles/common.yaml` `zones:` — каждая роль видит **только свою зону**:
оркестратор — только высокоуровневые решения (без механики инструментов);
conscience — то же, что оркестратор (это его зеркало); context_former — только
текущую задачу и то, что сам вытащил из кода; researcher — read-only факты;
coder — только реализацию данной задачи; tester — только тестирование
результата coder'а; executor — только runtime-запись атомарных шагов
замороженного плана. Если роли не хватает информации — она обязана
эскалировать вопрос вверх по цепочке (`escalation_chain`), а не додумывать
самой. Это ограничивает контекст каждого субагента до необходимого минимума и
не даёт роли выйти за пределы своей ответственности.

## 3. Режимы (`modes.yaml`)

### planning

Полная методика планирования через сервер plan-manager: HRS (человеко-читаемая
спецификация с бинарными параграфами) → MRS (машинная проекция: концепты +
типизированные связи) → GS/TS/AS (глобальный/тактический/атомарный шаг) →
каскад изменений → гейт (механические проверки) и скоринг (семантическая
оценка) → runtime-слой (execution_attempt/review_result/bug/todo).

- `always_read: [servers/planmgr.yaml]` — всегда читается при входе в режим:
  полный глоссарий терминов, 12-шаговая методология, обзор 27 командных семейств,
  таблица «блок → ops-файл».
- 10 ops-файлов покрывают блоки: `planmgr-authoring` (авторинг HRS/MRS/GS/TS/AS),
  `planmgr-cascade` (каскад + freeze/unfreeze), `planmgr-verification` (гейт,
  скоринг, weak-check, аудит), `planmgr-graph` (порядок исполнения, волны,
  явные зависимости), `planmgr-prompt` (сборка контекст-блоков и промптов для
  исполнителя/ревьюера), `planmgr-runtime` (записи попыток исполнения,
  ревью, эскалаций, комментариев, model binding), `planmgr-bugs` (баг-цикл),
  `planmgr-todo` (todo-очередь), `planmgr-project` (привязка плана к проекту),
  `planmgr-export` (экспорт/импорт/снапшоты).

### analysis

Read-only инспекция кода: поиск, AST/структура/качество, опрос асинхронных
job'ов на code-analysis-сервере. Никакой мутации файлов.

- `always_read: [servers/ca.yaml]`.
- 3 ops-файла: `ca-search` (fulltext/BM25/семантика/grep),
  `ca-analysis` (AST/структура/качество — приоритет над поиском),
  `ca-jobs-queue` (опрос job_id из очереди).

### refactoring

Трассировка usages/dependencies/иерархии + редактирование файлов проекта через
связку CA + ai-editor + terminal.

- `always_read: [servers/ca.yaml, servers/editor.yaml]`.
- 6 ops-файлов: `ca-refactoring-support` (usages/deps/hierarchy),
  `editor-edit-lifecycle` (open/preview/edit/write/close существующего файла),
  `editor-new-file` (создание нового файла / .toml-обход),
  `editor-formats-addressing` (адресация правок по формату файла),
  `ca-git` (все git-операции на серверном проекте),
  `terminal-sandbox` (команды в песочнице).

### Как добавить свой режим

Добавьте в `modes.yaml` новую запись под `modes:` по образцу:

```yaml
  my_mode:
    aliases: [мой_режим]              # опционально — синонимы на естественном языке
    description: >-
      Короткое описание, что за работа и с какими серверами.
    always_read: [servers/<your_server>.yaml]   # карта(ы) сервера, читаемые ВСЕГДА при входе в режим
    triggers:
      - when: <условие первого использования конкретной группы команд>
        read: [ops/<your-ops-file>.yaml]
      # ... по одному триггеру на каждый функциональный блок
```

Правило то же, что и для существующих режимов: `always_read` — читается сразу
при объявлении режима; каждый элемент `triggers` — только перед первым
фактическим использованием привязанной к нему группы команд.

## 4. Профили доступа к файлам (`roles/laws.yaml` → `variables.file_access`)

Профиль выставляется **пользователем/оркестратором один раз в начале сессии**
как значение переменной `variables.file_access` в `roles/laws.yaml` (по
умолчанию `mcp`). Он определяет, какими инструментами роли имеют право
касаться файлов проекта.

| Профиль | Редактор | Терминал | Поиск/анализ | Git |
|---|---|---|---|---|
| **mcp** (по умолчанию) | MCP ai-editor-сервер (`universal_file_*` lifecycle) | MCP terminal-сервер (sandbox / host_exec) | MCP code-analysis-сервер | MCP `ca git_*` команды |
| **local** | локальные инструменты агента (Write/Edit) | локальный bash | code-analysis-сервер (удалённый репозиторий + поиск/анализ) | `ca git_*` (CA выступает удалённым репо: push из локали / `ca git_pull_safe` на сервере) |
| **local-IDE** (гибрид) | ЧЕЛОВЕК в IDE (агенты предлагают патчи/диффы в отчётах, сами не редактируют) | не задействован агентами для правок кода | code-analysis-сервер | git снаружи агентов: push из локали человеком, `ca git_pull_safe` синхронизирует серверную копию |

### mcp — полностью удалённый (по умолчанию)

Все файловые операции — через связку MCP-серверов (CA + editor + terminal).
Локальные инструменты запрещены для файлов проекта (`mcp_only`), кроме
единственного исключения — чтения ролевых файлов (`roles/*`). Это профиль по
умолчанию во всех примерах и `ops/*.yaml` этого шаблона.

### local

Редактор = локальные инструменты агента, терминал = локальный bash. CA при
этом продолжает играть роль «удалённого репозитория» и источника поиска/анализа
— серверные git-операции идут через `ca git_*`, а не напрямую. Действует
**закон commit-after-edit**: после КАЖДОЙ локальной правки файла —
немедленный git commit, без накопления незакоммиченных изменений (см.
`roles/laws.yaml` → `local_mode`, `roles/tooling-rules.yaml` → `cas_rules.git_via_ca`).

### local-IDE (гибрид)

Файлы физически лежат локально, но их правит **человек** средствами своей
IDE — агенты код не редактируют напрямую: любой агент, дошедший до
необходимости правки, вместо `Write/Edit` формулирует патч/дифф и кладёт его в
свой отчёт наверх (`context_former`/`orchestrator`/пользователю), решение о
применении — за человеком. Поиск и анализ по-прежнему идут на
code-analysis-сервер (индекс серверной копии проекта). Синхронизация локальной
копии и CA — через git: человек делает push из локали, CA подтягивает через
`git_pull_safe` на сервере (или наоборот, в зависимости от того, где truth).

В этом профиле остаются активными:
- `escalation_first_error` и `reproduce_before_claiming` — без изменений, это
  универсальные законы, не завязанные на способ доступа к файлам;
- read-only использование `roles/*` и MCP CA для поиска/анализа — не отключается;
- `mcp_only` фактически не блокирует ничего для файлового ЧТЕНИЯ (диагностика,
  цитирование кода в отчёте разрешены), но НЕ применяется к записи — записи в
  этом профиле не существует со стороны агентов вовсе;
- editor-lifecycle (`ops/editor-edit-lifecycle.yaml` и т.п.) в этом профиле
  просто не используется — агенты не открывают/не коммитят файлы через
  ai-editor, только читают/анализируют и предлагают правки текстом.

Объявление профиля: пользователь/оркестратор фиксирует нужное значение
`variables.file_access` в `roles/laws.yaml` **до** первого спавна субагентов на
задачу — сменить профиль в середине сессии не запрещено технически, но должно
быть явным решением оркестратора, а не молчаливым дрейфом.

### editor_gate_no_bypass — HARD, не отключается никогда

Отдельно от профиля файлового доступа: если редактор отклоняет запись из-за
ошибок QA-гейта (mypy/flake8/валидация) — это HARD-правило, которое не
выключается **ни одной переменной, ни в одном профиле**. Правильная реакция:
стоп → анализ причины → исправление → повтор. Запрещено ослаблять гейт
(`skip_quality_tools`, `validate_syntax_only` и подобные обходы) — нулевая
терпимость гейта к ошибкам — осознанный дизайн против размножения моделью
мусорного кода; если гейт даёт ложный срабатывание — это баг инструмента,
чинится инструмент, обход всё равно запрещён (`roles/laws.yaml` →
`editor_gate_no_bypass`).

## 5. Чеклист адаптации под ваш проект

**(а) Заполните плейсхолдеры.** Полный список — ищите `grep -rn '{{' .` в этом
каталоге:

```bash
# 1) все YAML валидны
python3 -c "import yaml,glob; [yaml.safe_load(open(f)) for f in glob.glob('**/*.yaml', recursive=True)]" && echo OK

# 2) все bare-имена файлов (common.yaml, laws.yaml, tooling.yaml, tooling-rules.yaml,
#    modes.yaml и т.д.), упомянутые в reads_first/always_read/триггерах, физически
#    существуют где-то в дереве архитектуры (bare-имя -> файл в roles/, ops/, servers/
#    или в корне; путь вида ops/foo.yaml или servers/foo.yaml -> существует от корня)
python3 -c "
import glob, re, os
names = {os.path.basename(f) for f in glob.glob('**/*.yaml', recursive=True)}
missing = []
for f in glob.glob('**/*.yaml', recursive=True):
    for m in re.findall(r'[\w./-]+\.yaml', open(f).read()):
        if m.startswith('{{') or m.startswith('/') or 'PROJECT_STANDARDS_DIR' in m:
            continue
        base = os.path.basename(m)
        root_path = os.path.exists(m)
        same_dir = os.path.exists(os.path.join(os.path.dirname(f), m))
        by_name = base in names
        if not (root_path or same_dir or by_name):
            missing.append((f, m))
print('missing:', missing or 'none')
"
```

## 6. Принципы, которые нельзя ломать

- **Zero-tolerance QA-гейт.** Гейт редактора (`editor_gate_no_bypass`) не
  ослабляется никогда и ни в каком профиле доступа. Это осознанный дизайн
  против того, чтобы модели тихо размножали технический долг «лишь бы прошло
  commit» — единственный правильный ответ на красный гейт — почини причину.
- **Независимая верификация блокирующих claim'ов.** Если субагент заявляет
  что-то, что останавливает работу («API отсутствует», «сервер лежит», «нет
  синглтона») — прежде чем на это реагировать (эскалация, смена курса),
  роль, способная спавнить субагентов, обязана перепроверить claim ДРУГОЙ
  моделью в read-only режиме. Не подтвердилось — считать глюком заявившего и
  продолжать без эскалации; подтвердилось — действовать на основе реального
  дефекта, отличая дефект плана от дефекта кода (`verify_claims_independent_model`).
- **Воспроизведение прежде объявления поломки.** Никогда не объявлять
  инструмент сломанным и не заводить баг по одному-единственному сбою —
  среда флапает. Сначала — help/info, затем чистое повторение; баг — только
  если сбой воспроизводится стабильно (`reproduce_before_claiming`).
- **Стоп на первой инфраструктурной ошибке.** Без ретраев, без переключения
  серверов/сессий, без обходных путей — эскалация с точным вызовом, дословной
  ошибкой и конкретным вопросом (`escalation_first_error`).
- **Git на сервере — только через CA.** Ни при каком профиле файлового
  доступа серверные git-операции не делаются напрямую по ssh/консоли без явного
  разрешения пользователя — только через `ca git_*` команды
  (`cas_rules.git_via_ca` в `roles/tooling-rules.yaml`).

## Дисциплина веток (закон `branch_discipline`)

Трёхветочная схема разводит два потока правок (агенты на сервере / человек+агент локально)
и держит `main` всегда деплой-пригодным:

| Ветка   | Кто пишет                          | Роль |
|---------|------------------------------------|------|
| `cas`   | агенты на сервере (авто-коммиты редактора, CA `git_commit`) | рабочая ветка серверного дерева |
| `local` | локальная работа (человек в IDE / агент в local-режиме)     | рабочая ветка локального чекаута |
| `main`  | никто напрямую                     | только интеграция и деплой |

Потоки:

```
local ──merge──▶ main ◀──squash-merge── cas
                  │
                  ▼ (деплой всегда из main)
   build image → deploy → главный тест-скрипт по задеплоенному
```

Правила:
1. `cas → main` — **только squash** (одна поставка = один осмысленный коммит на main;
   микро-история остаётся в `cas`), выполняет только оркестратор и только после
   прохождения `delivery_checks`.
2. После каждого обновления `main` — **обязательный** `main → cas` (через CA `git_merge`)
   до продолжения работы агентов.
3. **`main → local` pull разрешён только после состоявшегося деплоя** — локаль забирает
   исключительно деплой-проверенные состояния `main`.
4. Серверный `main` — защищённая ветка: агент физически не мерджит и не пушит в неё
   (protected-branch policy у CA `git_push`).
5. При грязном серверном дереве синхронизацию делать через `git_pull_safe`
   (stash → pull → apply с диагностикой), а не голым pull.
6. **Симметрия локального режима**: при работе локально принцип тот же, меняющаяся
   ветка — `local`; merge `local → main` и push/pull выполняются **сразу после
   завершения каждого задания или шага**, без накопления.
