## 1. Главная идея: план как API, растянутое во времени

Замороженный план — это **статичный шаблон**. Его промпты — не готовые письма моделям,
а **сигнатуры вызовов**: что должно быть сделано, кем (роль), чем (toolset), на каком
уровне модели. Как у обычного API есть статическая сигнатура и динамические аргументы,
так и здесь: статика замораживается при авторинге, а всё динамическое подмешивается в
момент исполнения.

При исполнении каждый шаг проходит конвейер:

```
замороженный промпт (статичный шаблон, hash-верифицирован)
        │
        ▼
модель-ПОДГОТОВИТЕЛЬ КОНТЕКСТА  ← подмешивает динамику:
        │                          • срезы диалога (диалог-сервис)
        │                          • выдержки из базы знаний (doc-store)
        │                          • reality-supplement (текущее состояние репо)
        ▼
КООРДИНАТОР отправляет готовый промпт исполнителю (батч-вызов)
        │
        ▼
конверт-ответ: RESULT | ESCALATION | TOOL_CALL
```

Иерархия интеллекта по стоимости: умная модель — неспешная оффлайн-оптимизация контекста;
средние модели — владельцы/решения; дешёвые — писатели; координатор — «тактовый
генератор», детерминированный настолько, что его потянет 4-битная llama.

## 2. Участники

| Участник | Природа | Обязанности |
|---|---|---|
| Координатор | детерминированный процесс за рабочим местом | диспетчеризация волн, обработка конвертов, трансляция tool-call, маршрутизация эскалаций, коммиты |
| Подготовитель контекста | модель | обогащение статичного промпта динамикой перед вызовом |
| Naked-модель (писатель/верификатор) | смертный батч-вызов | получила промпт → вернула конверт → умерла; без окружения и инструментов |
| Владелец (T/G/план) | роль + механически собранный контекст, НЕ живая сессия | решения по эскалациям в рамках жёсткой типологии |
| Пользователь | терминал эскалаций | решения, которые обязаны дойти до человека |

Ключевой принцип: **владение переживает смертность агентов** — владелец это РОЛЬ +
КОНТЕКСТ (собирается командой планировщика за один вызов), а не запущенный процесс.

## 3. Онлайн / оффлайн (workplace binding)

- **ОНЛАЙН** (интерактив): каждая модель работает ЗА рабочим местом — прямой доступ к
  инструментам, эмуляция не нужна. Планировщик — один из инструментов на столе.
- **ОФФЛАЙН** (батч): за рабочим местом сидит ТОЛЬКО координатор. Батч-модели голые:
  промпт на входе, конверт на выходе. Весь доступ к инструментам — по доверенности,
  через координатора (см. §4).

Один и тот же контракт «оснащение рабочего места для шага» (сущности CR-5a) потребляется
напрямую в онлайне и по доверенности в оффлайне.

## 4. Трансляция вызова инструментов (акцент 2026-07-20)

Батч-модель не может вызвать инструмент сама — но может **попросить**. Координатор
воспроизводит агентный tool-use цикл поверх любого транспорта:

```
координатор                                батч-модель
    │  промпт + список инструментов            │
    │  (toolset шага, с инструкциями)          │
    ├──────────────────────────────────────────▶
    │                                          │ думает
    │            конверт TOOL_CALL(name, args) │
    ◀──────────────────────────────────────────┤  (вызов умер)
    │ 1. валидация: name ∈ whitelist toolset?  │
    │ 2. маршрут из сущности tool:             │
    │    server + command + PINNED options     │
    │    (args модели подмешиваются ПОД        │
    │     закреплённые — sandbox)              │
    │ 3. исполнение через MCP-прокси           │
    │ 4. результат → в диалог-сервис           │
    │                                          │
    │  СЛЕДУЮЩИЙ промпт = транскрипт           │
    │  (задача + пины + tool_result)           │
    ├──────────────────────────────────────────▶
    │                ... цикл ...              │
    ◀── конверт RESULT (или ESCALATION) ───────┤
```

Существенные свойства цикла:

1. **Каждый ход — новый батч-вызов.** Состояние диалога держит НЕ модель и НЕ
   координатор в памяти, а диалог-сервис: координатор аппендит каждое сообщение
   (задача, tool_call, tool_result, эскалация, решение) и перед следующим ходом
   запрашивает срез. Ответ инструмента модель получает **со следующим промптом** —
   это и есть «растяжка по времени».
2. **Whitelist**: инструмент вне toolset шага → отказ/эскалация, не исполнение.
3. **Pinned options = sandbox**: закреплённые в сущности tool опции (project_id,
   префикс путей, лимиты) модель перебить не может.
4. **Лимиты цикла**: max итераций на шаг, per-call timeout (поля invocation profile);
   ошибка инструмента возвращается модели как данные, а не роняет шаг.
5. **Идемпотентность** (открытый вопрос, зафиксированы ограничения): журнал ключей
   tool-call у координатора; ретрай после потери ответа не должен повторно исполнить
   мутирующий вызов. Несколько tool_call в одном ответе — исполняются строго
   последовательно (v1 без параллели).
6. **Безопасность**: терминала нет ни у одной naked-модели (и у оффлайн-координатора);
   секреты вырезаются из результатов до попадания в транскрипт; инструкции — только из
   плана, результаты инструментов — данные (анти-инъекция); большие результаты
   пагинируются координатором.

**Что ещё продумать** (по вашей реплике): протокол нормализации конвертов между
провайдерами (родные форматы tool-calling у всех разные — слой model-access-core),
версия схемы конверта, и точная граница «что подготовитель контекста подмешивает сам
vs что подшивает координатор механически».

## 5. Эскалационная шина

Эскалация — не «отправка сообщения», а **конверт вместо результата**. Координатор,
увидев ESCALATION: не записывает результат → командой планировщика собирает контекст
владельца → батч-вызов владельцу → решение возвращается вниз (перезапуск исходного
агента: замороженный промпт + решение) или уходит выше рекурсивно.
Лестница: writer → ts_owner → gs_owner → orchestrator → **user** (терминал).

Решения пользователя 2026-07-20 (стоящие):

- **Типология полномочий владельца** (закрытый список): interpret / supplement
  (дополнение диспетчерского контекста, класс reality-supplement) / needs-plan-change
  (→ каскад, наверх) / abort. Замороженную истину владелец не правит НИКОГДА.
- **Пинг-понг**: max 2 раунда на (шаг, вопрос); третья эскалация того же вопроса —
  принудительно на уровень выше; терминал без решения → шаг паркуется.
- **Ночь**: терминальная (пользовательская) эскалация **паркует ВСЮ волну**;
  таймаут вызова владельца → парковка, никогда не abort.

Шина = хранилище записей с приоритетами (nice) + свип координатора: открытые эскалации
обрабатываются раньше выдачи новой работы; открытая эскалация блокирует свой шаг и его
поддерево.

## 6. Диалог-сервис (отдельный проект)

Внешнее хранилище цепочек диалога. Правила детерминизма: **номера сообщений
авторитетны**, семантическая релевантность — только советник; каждый срез обязательно
содержит **пины** — задачу шага, все решения по его эскалациям, последние N сообщений.
Провенанс: ссылка транскрипт ↔ execution_attempt (поле резервируется в CR-5a).

## 7. Файлы, коммиты, восстановление

Решение пользователя 2026-07-20: **агенты распределяются по файлам** (один писатель на
файл); принцип — «сделал работу → её приняли → коммит», даже если дальше будет ещё
правка (восстановимость промежуточных состояний); при работе нескольких сессий с одним
файлом коммит обязателен.

Оффлайн-проекция: координатор коммитит после каждого **принятого** AS-шага в ветку
волны; слияние в main — только L1/пользователь после утреннего ревью; падение посреди
цикла → redo от последнего коммита; деструктивный git в общем дереве запрещён.
Восстановление координатора после краха: lease/heartbeat волны + повторный вход по
транскриптам (протокол — в плане координатора, не здесь).

## 8. Слой данных CR-5a (текущий план, 6d12ea18)

Что именно поставляет ЭТОТ план (только данные и команды, никакого рантайма):

- сущности: **tool** (server+command+pinned options), **toolset**, **role** (апгрейд
  enum→таблица), **provider** (тип cloud_api|self_hosted, rented-флаг, статус
  активности — ось переключения), **model** (имя+провайдер+**LEVEL**+окно+класс
  стоимости+режим interactive|batch);
- **role↔model** по уровням, назначение ВРУЧНУЮ (рейтинги отложены);
- **per-step назначение** роли и toolset (та же 6-уровневая лестница, что у model_binding);
- **invocation profile** — информационные характеристики вызова (генерационные
  параметры, таймауты, ретраи, лимиты tool-цикла, бюджеты, режим, ссылка на диалог);
- **эскалационные данные**: адресат, цепочка, схемы конвертов RESULT|ESCALATION|TOOL_CALL,
  приоритет, блокировка поддерева + **команда сборки контекста владельца**;
- **типизированный учёт**: {tokens_in, tokens_out, provider, model, wall_ms,
  cost_estimate?} в execution_attempt + структурный changed_by (role@step_path@attempt);
- политика эскалаций (§5) как HRS-параграф.

## 10. Открытые вопросы (остаются на продумывание)

1. Нормализация родных tool-calling форматов провайдеров → единый конверт
   (слой model-access-core): где именно граница адаптера, версия схемы конверта.
2. Точный протокол lease/heartbeat/takeover координатора и детект полуисполненных
   tool-call при рекавери (связан с журналом идемпотентности).
3. Верификация оффлайн: verifier-роль вызывается как владелец и пишет review_result —
   кто запускает верификационные команды при полностью батчевом цикле.
4. Механика KB-выборки (по id / поиск / embedding-релевантность) — грунтовать по
   реальному doc-store при авторинге соответствующего плана.
5. Граница обязанностей подготовителя контекста vs механической подшивки координатора
   (ваша реплика 2026-07-20 про «подмешивание кучи всего» — см. §1 и §4).
